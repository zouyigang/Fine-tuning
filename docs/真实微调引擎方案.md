# 真实微调引擎实施方案（本机 NVIDIA GPU + LLaMA-Factory）

> 状态：方案已定稿，决策点 + 版本已锁定，待开 M1。
> 关联：替换 `server/app/services/trainer.py` 的模拟训练。其余四大模块（数据集/评估/版本/配置）web 闭环已完成。

## 0. 决策（已拍板）

| # | 决策 | 结论 |
|---|------|------|
| 1 | 基础模型来源 | 本机**离线**，HF 权重放 `MODELS_DIR`；当前以 **Qwen3 系列**为主，架构层通用支持其他大模型 |
| 2 | 数据集格式 | M1 **只支持标准格式**（alpaca / sharegpt 直通），业务原始 schema 映射后续做 |
| 3 | Docker | **做 CUDA 镜像**（`real` 模式）；`sim` 模式保留 slim 镜像供无 GPU 环境/CI 演示 |
| 4 | M1 验收口径 | 小模型（**Qwen3-0.6B**）+ 小数据集跑通一轮，监控页见**真实 loss 下降**，训练完**自动生成 `model_version`** |
| 5 | 本机环境 | **保持不动**（torch 2.11.0+cu130 已能跑），不升级 |

## 1. 环境与版本锁定（已核实，2026-06-23）

**Docker 镜像锁定版本**（与本机解耦，独立 pin 以可复现）：

- torch / **torchaudio / torchvision** 三者 **2.11.0+cu130**（CUDA 13.0，支持 Blackwell/sm_120；与本机已验证配置一致）
- LLaMA-Factory **0.9.5**（最新正式版）
- ⚠️ **必须三件套同版本同 CUDA 源**：llamafactory 启动时无条件 `import torchaudio`，torch 与 torchaudio 的 CUDA 版本不一致会在 import 阶段直接 `RuntimeError` 退出。torchaudio 目前最高只到 **2.11.0**，且 cu132 源**没有** torchaudio——这正是 torch 2.12.1+cu132 方案跑不起来的原因，故回退到完整的 2.11.0+cu130 套件。
- 关键认知：**实际生效的 CUDA/cuDNN 由 torch wheel 决定**（cu130 自带 CUDA 13.0 + 捆绑 cuDNN），基础镜像的系统 CUDA/cuDNN 对「pip 装 torch 跑 LoRA」是冗余的。除非源码编译 torch/flash-attn，否则无需在镜像里单独装 CUDA/cuDNN。
- **宿主机只需安装最新 N 卡驱动** + nvidia-container-toolkit；driver 向前兼容（新驱动跑任何 ≤ 它的 CUDA）。

推荐 Dockerfile（`real` 模式，最简最小）：
```dockerfile
FROM python:3.12-slim
RUN pip install torch==2.11.0 torchaudio==2.11.0 torchvision==0.26.0 --index-url https://download.pytorch.org/whl/cu130
RUN pip install llamafactory==0.9.5 -r requirements.txt
# compose: gpus: all ；宿主：最新驱动 + nvidia-container-toolkit
```
（仅当未来要编译 flash-attn/deepspeed 才改用 `nvidia/cuda:13.2.x-cudnn-devel-ubuntu22.04` 基础镜像。）

## 2. 总体架构（进程模型）

```
FastAPI 进程
  └─ TrainingManager（单例：并发闸 + 任务队列）
       ├─ 队列 worker：按可用 GPU 数取 pending 任务
       └─ 每个运行任务 →
            subprocess.Popen(llamafactory-cli train <task.yaml>)  [CUDA_VISIBLE_DEVICES=N]
            + 监控线程：tail output_dir/trainer_log.jsonl → 写 train_metric
                        tail stdout/stderr               → 写 train_log
            进程退出 → 置 success/failed + 自动建 model_version
```

**为什么用 subprocess 调 `llamafactory-cli` 而非 import 调库**：训练重 GPU、长时阻塞、可能 OOM/段错误；独立进程崩溃不拖垮 API，可单独设 `CUDA_VISIBLE_DEVICES`、可被 kill（实现停止/重试），CLI 是 LF 官方稳定入口。

## 3. LLaMA-Factory 集成

- 每任务生成 YAML：`storage/runs/{task_id}/train.yaml`，字段由超参映射：
  `stage: sft`、`finetuning_type: lora|qlora|full`、`model_name_or_path`(指向 `MODELS_DIR` 离线权重)、`dataset`、`learning_rate`、`per_device_train_batch_size`、`num_train_epochs`、`lora_rank/lora_target`、`output_dir`、`logging_steps`、`fp16/bf16`、`quantization_bit`(QLoRA=4)。
- 启动：`llamafactory-cli train <yaml>`，注入 `CUDA_VISIBLE_DEVICES`。
- 指标回流走 LF 产物 `output_dir/trainer_log.jsonl`（每 `logging_steps` 一行 `loss/lr/epoch/step`），比解析 stdout 稳。

## 4. 数据流（M1 标准格式直通）

- 上传 `dataset_file` 落在 `storage/datasets/`。训练前 converter 注册到 LF `data/dataset_info.json`。
- M1 仅支持：① 已是 alpaca/sharegpt 格式；② `{prompt,response}` 简单映射。其余业务 schema 映射后续单独做。

## 5. 数据模型改动（用 Alembic 迁移，不靠 create_all）

`train_task` 增列：`method`、`hyperparams`(JSON)、`base_model_path`、`output_dir`、`pid`、`error_msg`、`model_version_id`、`finished_at`。
新增表 `task_artifact`：产物清单（adapter/merged 权重路径、大小）。
`model_version` 增 `task_id`：建立「任务 → 产出模型」关联。

> create_all 不会给既有表加列 → 必须写 Alembic revision（autogenerate 已就绪）。

## 6. 训练生命周期

`pending → running → success | failed | stopped`
- 创建任务：**改为默认 `pending` 入队**（现状是「创建即 running」）。
- worker 有空闲 GPU → 拉起 `running`，记 `pid`。
- 停止：kill → `stopped`；重试：复用配置重新入队。
- 成功：解析最终指标 → 建 `model_version`（status=`evaluating`，f1 由评估模块算）。

## 7. 真实指标 / 日志回流（替换 trainer.py）

- 由 `ENGINE_MODE` 控制：`sim`=保留现状（演示任务 id=1 常驻造假曲线）；`real`=关闭模拟，只跑真实任务。
- 监控线程增量读 `trainer_log.jsonl` → 写 `train_metric`；GPU 利用率用 `pynvml`/`nvidia-smi` 真实采样，替换现在的 `random`。
- `running_payload`/`training_curve` 读逻辑基本不动（已从 train_metric 读），数据变真。

## 8. 训练完成 → 自动建 model_version

- LoRA 可选 `llamafactory-cli export` 合并出 merged 权重到 `storage/models/`，写 `task_artifact`。
- 建 `model_version`（name=任务名+base、version 顺延、dataset、status=`evaluating`、trainAt=now、creator、task_id）。
- 后续走已打通的「评估→上线→回滚」闭环。

## 9. GPU / 并发调度

- `pynvml` 探测 GPU 数；并发闸 = 可用 GPU 数（可配 `MAX_CONCURRENT_TRAINS`）。
- 每任务独占 1 卡（`CUDA_VISIBLE_DEVICES`）；超额任务留 `pending` 排队（对接现有 `schedule_item`）。
- OOM → 捕获退出码，置 `failed`+`error_msg`，释放 GPU。
- ⚠️ 本机单卡 16GB：Qwen3-0.6B/1.7B/4B 可 LoRA/QLoRA；8B 建 QLoRA(4bit)；14B+ 谨慎。M1 用 0.6B。

## 10. 超参模板 / 配置应用到任务（补现缺口）

- `create.vue` 已采集 method/lr/batchSize/epochs/optimizer 但 `createTask` 丢弃 → 改为**透传**存入 `train_task.hyperparams`。
- 「超参模板」下拉接真实 `getHyperTemplates`，选中填充表单。
- `hyperparams.vue` 的「应用到任务」即有落点（写目标任务 hyperparams）。

## 11. 新增配置项（settings）

`ENGINE_MODE`(sim/real)、`LLAMAFACTORY_BIN`、`MODELS_DIR`、`RUNS_DIR`、`MAX_CONCURRENT_TRAINS`。

## 12. 里程碑

- **M1** 单卡最小闭环：超参透传 + Alembic 加列 → YAML 生成 → subprocess 跑通 Qwen3-0.6B LoRA → 真实指标/日志回流 → 完成建 model_version。（先证明链路）
- **M2** 数据 converter：业务数据集 → LF 格式 + dataset_info 注册。
- **M3** 控制面：停止/重试、GPU 探测与并发闸、队列对接。
- **M4** 产物：adapter 合并/导出、`task_artifact`、串联导出/部署模块。
- **M5** 收尾：真实 GPU 采样替换 random、错误/OOM 处理、CUDA 镜像、`ENGINE_MODE` 切换、文档。

## 13. 风险与对策

| 风险 | 对策 |
|------|------|
| 业务 schema → 指令样本映射不明确 | M1 只支持标准格式，映射规则单列 |
| 显存/OOM 拖垮机器 | 独立进程 + 并发闸 + QLoRA 默认 + 捕获退出码 |
| Docker 无 CUDA | `real` 用 CUDA 镜像（torch cu130，宿主仅驱动）；`sim` 用 slim |
| 模拟与真实混淆 | `ENGINE_MODE` 一刀切，演示任务仅 sim 下常驻 |
| 本机单卡 16GB 受限 | M1 用 Qwen3-0.6B；大模型走 QLoRA |

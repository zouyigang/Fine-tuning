# 前端改造方案 · 消除演示数据 + 补全死按钮

> 状态：方案待评审，分三阶段可独立排期。
> 背景：前端五大模块写操作已真实落库，但盘查发现仍有 **6 处页面在展示演示/静态数据**、**20 个按钮点击无响应**。本方案给出逐项改造路径。
> 关联：与 `docs/真实微调引擎方案.md` 解耦，可并行推进；评估指标真实化（路线 B）依赖二期评估引擎。

## 0. 总原则

| # | 约定 | 说明 |
|---|------|------|
| 1 | 前端零侵入 | 静态数据页**只改 `src/api/modules/*.js` 与后端**，不动 `.vue`；仅死按钮才改 `.vue` |
| 2 | 统一响应 | 新增接口沿用 `{code,data,message}`，分页 `{list,total,page,pageSize}` |
| 3 | 审计与鉴权 | 新增写操作挂 oplog 中间件（补 `_RULES`），高风险写操作按需补 `deps.py` 的 `_RBAC_RULES` |
| 4 | 不再造假 | 真实来源缺失的字段给**降级策略**（聚合可得部分 + 占位），不再用 `Math.random` / 硬编码常量 |

## 2. 阶段一　静态数据真实化（约 2–3 天）　✅ 已完成（2026-06-24）

> 落地：新增 `routers/screen.py` + `crud/screen.py`（`GET /screen/overview` 全量聚合）；
> `crud/dashboard.py` 趋势改真实聚合（共用 `screen.task_trend`）；评估三页走路线 A，
> 新增 `eval_metric` / `eval_per_class` / `benchmark_result` / `scene_case` 四表
> （models + seed `ensure_eval_aggregates` 幂等 + Alembic `0003_eval_aggregates`），
> crud 改读表；报告 F1 改 `_model_f1` 从真实模型版本派生。前端仅 `screen.js` 换真实接口。
> 端到端 TestClient 冒烟：5 接口 code=0 + 报告 F1 真实。

### 2.1 数据可视化大屏（最优先）

**做法**：新增 `server/app/routers/screen.py` + `crud/screen.py`，提供 `GET /screen/overview`，**严格按 `screen/index.vue` 现有数据结构**返回。前端只把 `screen.js` 改成：

```js
import service from '@/api/request'
export function getScreenData() {
  return service.get('/screen/overview')
}
```

页面与 8s 轮询逻辑一字不动。逐字段聚合来源：

| 字段 | 真实来源 | 降级策略 |
|---|---|---|
| `headline` 数据集/任务/在线模型/样本量 | `count(Dataset)` / `count(TrainTask)` / `ModelVersion.status='online'` / `sum(Dataset.total)` | — |
| `taskTrend` 近 14 天 | `TrainTask` 按 `createdAt`/`finishedAt` 按日聚合 | — |
| `datasetDist` | `Dataset` group by `type` | — |
| `logs` 实时流水 | **直接读 `operation_log` 最近 N 条**（已是真实数据） | — |
| `tasks` 运行状态 | `TrainTask` 取 running/queued | — |
| `release` 上线流水 | `gray_release` + `release_history` | 空列表 |
| `gpuNodes` | 真实引擎 `engine` pynvml 采样 / `ClusterInfo` | 单卡/占位 |
| `annotation` 各警种进度 | `annotation_task` 按部门聚合 | 现有任务均值 |
| `radar` / `spotlight` 准确率 | `EvalReport.f1` / `ReviewSample` 准确率聚合 | 取最近报告 |

> `taskTrend` 聚合函数与 2.2 共用，避免重复实现。

### 2.2 工作台趋势图

`crud/dashboard.py` 中把趋势的 `random.randint` 改为 `TrainTask` 按日期真实聚合（复用 2.1 的查询）。其余指标卡 / 模型分布 / 最近任务 / 待办已是真聚合，不动。

### 2.3 评估三页 + 报告 F1

**根因**：无真实评估引擎，`crud/evaluation.py` 直接返回硬编码常量。两条路线：

- **路线 A（轻量，~1 天，先做）**：常量迁入 DB 表 `eval_metric` / `benchmark_result` / `scene_case` 并 seed，crud 改为读表，可按 `model_type`/`model_id` 区分。数据从此可维护、随模型增删，去掉「代码写死」性质。报告 F1 改取关联 `model_version` 指标或该模型复核准确率，删除 `random.uniform`。
- **路线 B（重量，二期）**：真实评估引擎加载 M4 合并出的 `merged` 权重，在标注测试集上推理算 P/R/F1，落**同一批表**，只换数据生产者。

> 阶段一落路线 A；路线 B 与评估引擎一起做。
